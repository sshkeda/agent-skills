Follow these repo rules in order. If rules conflict, the earlier rule wins.

1. Treat `~/.agent-skills/config.json` as the machine-local source of truth; `config.example.json` is only a bootstrap example.
2. Use `node bin/agent-skills.mjs check` after editing the registry, skill copies, patch manifest, or reconciler script.
3. `~/.pi/agent/skills` is generated by `agent-skills apply` as a real managed directory of symlinks.
4. Leave `npx skills`-managed skills as real directories in `~/.agents/skills`; import them by linking from `~/.pi/agent/skills`.
5. Put local/custom skill source directories in `localSkills`, but keep each skill in its canonical owning repo: a repo-specific skill belongs under that repo (for example `~/gh/example-product/skills/example-product`), and a standalone skill belongs in its own repo. Do not add unrelated domain skill source files under this registry repo merely to make them discoverable.
6. Do not manage Pi packages or `~/.pi/agent/settings.json` from this repo.
7. If `claudeSkillsDir` is set, `agent-skills apply` also keeps matching symlinks in that directory (default usage: `~/.claude/skills` for the pi-claude-code rework). It tracks the names it created in `<claudeSkillsDir>/.agent-skills.json` and never touches entries it did not create — Claude Code's own skills must survive every apply.
8. If `codexSkillsDir` is set, treat it like `claudeSkillsDir`: add/remove only state-tracked symlinks and leave Codex-owned entries, including `.system`, alone.
9. Skills under this repo are limited to the registry's own skill or explicitly mirrored/maintained skills such as `yc-office-hours`; put local edits to mirrored skills in `skill-patches.json`, then run `node bin/agent-skills.mjs patch --check` and `node bin/agent-skills.mjs update --check --skip-npx`; do not hand-edit mirrored skill files without adding a replayable patch.
10. Never hand-symlink a skill into a generated skills directory (`~/.pi/agent/skills`, `~/.claude/skills`, `~/.codex/skills`, `~/.agents/skills`); register it in `localSkills` and run `agent-skills apply`. `check` fails on unmanaged symlinks in those directories; harness-owned real directories stay untouched.
11. Forward-test registry and skill changes with `agent-dogfeed`: probes resolve `--skill <name>` from the generated `~/.codex/skills` / `~/.claude/skills`, so a skill must be applied here before a probe can load it by name.
